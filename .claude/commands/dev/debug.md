# /debug <报错描述>

**铁律：未找到根因不能修。同一修复 3 次仍失败 → 停止，质疑架构。**

## 流程

### 1. 加载领域知识
使用 `debug` 技能获取：常见报错速查、3次即停判定标准、图形调用链追踪参数、gen-tests-rules.json 字段 schema、Bug 反哺决策矩阵。

### 2. 诊断
读报错 → 定位文件+行号 → 分析类型 → 检查 `git diff` 近期改动。

### 3. 图形调用链追踪（可选增强）
使用技能中的图形调用链参数调用 MCP graph tools。
- 如果图形可用：自动追踪上游调用者 + 下游被调用者 + 受影响执行流 + 测试覆盖 + 影响半径
- 如果图形不可用：继续手动 Read 上下文

### 4. 判断是否需要 orca-review
诊断出根因和修复方案后，按分级判定：
- **轻量级（文案/样式/注释修复）** → 跳过，直接进入步骤 5
- **标准级及以上（逻辑修复/代码改动）** → 触发 orca-review，评审修复方案后再进入步骤 5
  - Orca 可用 → 派 Claude 评审根因+修复方案 → 吸收意见
  - Orca 不可用 → 标注"外部评审未完成"，展示修复方案给用户确认。**禁止手工自审替代**

### 5. 最小修复
一次只改一个变量。

### 6. 验证
运行 `/verify`。

### 7. 3次即停
同一问题修 3 次还失败 → 告知用户疑是架构问题。

### 8. 修复完成后的反哺
分析 Bug 根因，判断 gen-tests 是否漏了此类场景。如能通过规则改进覆盖，询问用户是否加入推导规则。不能覆盖的（架构/环境问题）记录到 learnings 即可。
