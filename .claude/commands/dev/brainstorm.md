# /brainstorm <需求描述>

在写任何代码之前，先和用户聊清楚需求。**不允许跳过这步直接写代码。**

## 流程

### 1. 理解背景
- 给哪个项目做的？（未指定则先问，用 /list-projects 查看已有项目）
- 解决什么业务问题？
- 期望的最终效果？

### 2. 确认范围
复述理解，向用户确认：要做哪些、不做哪些、有没有可参考的现有功能

### 3. 输出验收标准
用业务语言列出完成标准，如"订单列表页有个标记重点按钮，点击确认后状态变更"

### 4. 确认后进入 /plan
用户确认后，自动进入实施计划阶段。
