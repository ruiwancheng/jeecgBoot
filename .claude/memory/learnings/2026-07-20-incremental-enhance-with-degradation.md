# [2026-07-20] [架构] 增量增强+降级策略模式 — Harness安全演进的基石

## 触发条件
对已有工程体系做大规模增强（本次 20 项），需要保证不破坏现有功能。

## 模式定义

每项增强遵循统一公式：
```
新增可选能力 + 守卫条件 + 文档化降级 = 安全演进
```

### 守卫条件模板
```bash
if command -v orca &>/dev/null; then
  # Orca 增强路径
else
  # 降级路径（等价的标准 Bash）
fi
```

### Skill 降级模板
```markdown
## 某某增强（Orca 增强，推荐）
...增强内容...

> 降级策略：Orca 不可用时 → 回退到标准行为（XXX）。
```

## 本会话实例

| 增强 | 守卫 | 降级路径 |
|------|------|------|
| 工作树隔离 | `orca worktree create` 失败 | 直接在开发目录执行 |
| 编排 DAG | Orca orchestration 不可用 | 回退到进程内并行 |
| Orca 终端管理 | `command -v orca` | 标准 Bash 命令 |
| Orca Browser | Orca 不可用 | Playwright `--headed` |

## 为什么这很重要

1. **零风险演进** — 不影响不装 Orca 的客户端
2. **渐进激活** — 装 Orca 后自动获得增强，不装不受影响
3. **可测试** — 降级路径就是等价功能，本身就是"对照组"
4. **文档自包含** — 降级策略直接嵌入 Skill 文件，不需要额外文档

## 判断标准
问自己："如果 Orca 明天消失了，这个增强会变成死代码吗？"
- 是 → 重构
- 否（有降级）→ 通过

## 关键教训
- 不要在 Hook 脚本中硬依赖 Orca — 所有 `orca` 命令都要 `2>/dev/null`
- 降级策略不能是"功能丢失" — 必须是"等价路径"
- Skill 文件中的降级策略用 `> 降级策略：...` blockquote 格式统一展示
