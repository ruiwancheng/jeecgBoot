# [2026-07-24] [Harness] 规则条件盲区——"X模式免评"类规则系统性脆弱

## 触发条件
规则中定义"某模式下免评/豁免"，但该模式涵盖的场景范围比规则编写时想象的大。

## 案例
`plan.md` 原规则：
```
直接模式（仅限文案/注释/样式）：免评
```
隐含假设：直接模式 = 只做文案/注释/样式。实际：直接模式也可以做代码改动（用户说"确认"后 AI 直接写代码）。

**规则说"直接模式免评"→ AI 理解为"所有直接模式都免评"→ 代码改动在直接模式下系统性跳过 orca-review。**

## 根因模式
```
if (mode == X) 豁免   ← 规则写的
X 可以做 A 也可以做 B  ← 实际情况
A 应该豁免，B 不应该   ← 真实意图
规则没有区分 A 和 B    ← 盲区
```

## 处理方式
- **禁止模式级豁免**："直接模式免评"、"delegate 模式豁免" 等以模式为单位的豁免一律不写
- **改为操作级豁免**："文案/注释/样式改动免评"——不管什么模式，只看改动类型
- 修复写法：
  ```
  ❌ 直接模式免评
  ✅ 任何模式：代码改动 → 强制评审；文案/注释/样式 → 免评
  ```

## 历史同类
- `orca-review-false-sense`：标准级改动 → "提示用户确认"（档位空转）
- `audit-classification`：`status` Integer vs VARCHAR "跨模块统一=非问题"（忽略了同模块内不一致的情况）
- 三者共同模式：**条件粒度太粗，覆盖了不该覆盖的场景**
