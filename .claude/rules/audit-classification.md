# 审计反馈自动分类

收到铁拳团/服务端审计报告后，AI 自动对每条共识风险进行分类判定。

## 分类标准

| 类型 | 判定条件 | 处理方式 |
|------|----------|----------|
| **真实 bug** | 代码逻辑确实错误、字段不匹配、校验缺失 | 立即规划修复 |
| **项目模式** | 与仓库/客户/供应商等其他模块使用相同模式 | 记录为"非问题"，不修 |
| **未来需求** | 依赖未建成的模块（如"未校验下游订单"） | 记录为 TODO，Phase 2 处理 |
| **平台级** | 需要平台层统一修改（如@Log、@PermissionData） | 记录为平台改进项 |
| **误报** | 审计理解有误、代码已修复但审计未感知 | 直接忽略 |

## 判定流程

```
审计报告 → 逐条对照此表 → 自动分类 → 输出判定结论
├── 真实bug → 规划修复
├── 项目模式 → 标注"与xx模块一致"
├── 未来需求 → 标注 TODO-PHASE2
├── 平台级 → 标注 PLATFORM
└── 误报 → 标注原因
```

## 执行规则

AI 在收到审计报告后，必须：
1. 逐条对照本表进行分类
2. 对标记为"真实bug"的立即进入修复规划
3. 对其他类型的给出判定理由
4. 汇总结论供用户确认

## 自学习规则

每次审计修复完成后，AI 检查修复的问题是否是**新模式**（当前速查表中没有的条目）：
- 是 → 提议追加一行到上表 `常见模式速查表`，格式：`| {发现} | {真伪} | {说明} | ← {日期} {模块}审计 |`
- 否 → 不操作

追加前必须展示给用户确认，格式：
```
🔄 学习闭环提议：审计发现新模式
   问题：{描述}
   判定：{真实bug/项目模式/未来需求/平台级}
   建议追加到 audit-classification.md 常见模式速查表
   确认？(y/n)
```
