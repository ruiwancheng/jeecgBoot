# 审计反馈自动分类

收到铁拳团/服务端审计报告后，AI 自动对每条共识风险进行分类判定。

## 分类标准

| 类型 | 判定条件 | 处理方式 |
|------|----------|----------|
| **真实 bug** | 代码逻辑确实错误、字段不匹配、校验缺失 | 立即规划修复 |
| **项目模式** | 与仓库/客户/供应商等其他模块使用相同模式 | 记录为"非问题"，不修 |
| **未来需求** | 依赖未建成的模块（如"未校验下游订单"） | 记录为 TODO，Phase 2 处理 |
| **平台级** | 需要平台层统一修改（如@Log、@PermissionData） | 记录为平台改进项 |
| **误报** | 审计理解有误、代码已修复但审计未感知 | 直接忽略 |

## 常见模式速查表（基于历史审计积累）

| 审计发现 | 真实 bug？ | 说明 |
|----------|:--:|------|
| status字段类型不一致(Integer vs VARCHAR) | ❌ 项目模式 | 仓库/客户/库位都用Integer+yn字典，物料/价格一致 |
| 删除未校验下游引用(TODO PEND-003) | ❌ 未来需求 | 下游表未建成，Phase 2补充 |
| @PermissionData缺失 | ❌ 平台级 | 所有MES模块统一处理 |
| 审计日志@Log缺失 | ❌ 平台级 | 平台层统一规划 |
| 单位/规格规则未固化 | ❌ 未来需求 | Phase 2需求 |
| 软删除复活覆盖旧数据 | ❌ 项目模式 | 所有模块统一设计 |
| 导入大事务无分批 | ❌ 未来需求 | Phase 1数据量小，Phase 2优化 |
| **前端字段绑定错误(dictCode)** | ✅ **真实** | 功能错误，影响数据正确性 |
| **日期校验缺失(end<begin)** | ✅ **真实** | 参数校验缺口 |
| **业务规则缺失(重叠/必填)** | ✅ **真实** | 业务流程完整性 |
| **并发先查后写无锁** | ⚠️ 部分 | 已有DuplicateKeyException兜底，但无分布式锁 |
| **导出全量无上限** | ⚠️ 部分 | 已有queryAll上限但导出可进一步优化 |

## 判定流程

```
审计报告 → 逐条对照此表 → 自动分类 → 输出判定结论
├── 真实bug → 规划修复
├── 项目模式 → 标注"与xx模块一致"
├── 未来需求 → 标注 TODO-PHASE2
├── 平台级 → 标注 PLATFORM
└── 误报 → 标注原因
```

## 执行规则

AI 在收到审计报告后，必须：
1. 逐条对照本表进行分类
2. 对标记为"真实bug"的立即进入修复规划
3. 对其他类型的给出判定理由
4. 汇总结论供用户确认
