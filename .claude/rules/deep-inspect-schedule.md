# 深度巡检调度规则

## 提醒时机

### 会话启动提醒
每次新会话启动时，检查 `hermes/eagle-eye/.last-deep-inspect` 的时间戳：
- 距今 > 7 天 → 打印提醒："深度巡检已超过 7 天未执行，建议运行 /deep-inspect"
- 距今 > 14 天 → 打印提醒：含 ⚠️ 标记，优先级提高
- 文件不存在 → 视为从未执行，打印："尚未建立性能/视觉基线，建议运行 /deep-inspect 初始化"

### 部署前提醒
当用户触发部署流程时，检查最近一次深度巡检：
- 距今 > 7 天且本次变更含 Controller/Service/Entity 修改 → 强烈建议先巡检
- 本次变更含 ≥5 个文件 → 必须提醒

## 自动触发条件

以下情况 AI 应主动询问是否需要深度巡检：

1. `/test-all` 发现 API 通过率下降 > 5%
2. 连续 3 次以上测试失败（可能是不稳定测试或基础设施问题）
3. 新增了 Entity 字段（可能有数据库性能影响）
4. 新增了 Controller（新端点缺少性能基线）
5. 前端新增了页面组件（新页面缺少视觉基线和无障碍检查）

## 模块选择逻辑

按优先级排序：
1. 当前活跃项目（`features.json` 中标记的项目）
2. 最近 7 天有 git 变更的模块
3. 上次巡检发现的 P0/P1 问题未修复的模块
4. 测试通过率最低的模块

## 巡检历史

巡检记录维护在 `hermes/eagle-eye/reports/` 目录，按日期分组。
每次巡检后更新 `.last-deep-inspect` 时间戳。
