# 开发流程

```
/brainstorm → /plan → 用户确认 → 写代码 → gen-tests(自动) → 铁拳团审计(本地) 
    → 修复审计问题 → /verify → git commit + push → 部署 
    → 鹰眼团测试(半自动提示) → /done → 人工验收
```

## 新增机制（V3.0）

| 节点 | 触发 | 说明 |
|------|------|------|
| **gen-tests** | 写代码后**自动**触发 | 从 Controller 自动推导 API 测试用例 + E2E 测试骨架。开发者提测前自测边界场景 |
| **铁拳团审计** | commit 前**自动**触发（新增模块/重大重构） | 本地 10 人 AI 并行审计，代码级风险检测，发现共识问题必须先修复再提交 |
| **鹰眼团测试** | 部署后**半自动**提示 | deploy 完成后提示"运行鹰眼团测试？"，确认后执行 gen-tests → test-api → test-e2e |

> **为什么审计前置到 commit 前？** 供应商模块的开发验证：审计后置到部署后导致 2 轮额外部署。前置到 commit 前修完再提交，一次部署即可。

## 防失忆触发条件（强制）

**每完成一步，必须在回复末尾提示下一步是什么。** 具体触发规则：

| 当 AI 刚刚完成了 | 必须做 | 触发词 |
|------|------|------|
| 需求讨论（/brainstorm 结束） | 提示"确认无误就进入 /plan" | — |
| 实施计划（/plan 结束） | 提示"确认计划就进入写代码" | — |
| 写代码完成（最后一批文件写入后） | **自动 gen-tests，然后自动进入铁拳团审计** | git diff 有变更 + 本次写入过文件 |
| 铁拳团审计发现共识 P0 | **先修复再继续**，修复后重跑审计 | 共识 P0 > 0 |
| 铁拳团审计通过（无共识 P0） | **自动进入 /verify** | — |
| /verify 通过 | **自动 commit + push，然后提示部署** | git diff 有变更 |
| 部署完成 | **半自动提示"运行鹰眼团测试？"** | — |
| 鹰眼团测试通过 | 提示进入 /done | — |
| 测试有失败 | 进入 /debug，修复后重测 | 报错信息 |

> ⚠️ **最常见失忆点：写完代码直接结束，跳过了 gen-tests 和铁拳团审计。**
>
> ⚠️ **第二失忆点：审计修复后直接部署，跳过了鹰眼团测试。** 记忆规则已写入 `feedback-eagle-eye-after-audit.md`。

## 铁拳团审计触发条件

| 条件 | 审计？ | 说明 |
|------|:--:|------|
| 新增模块（新 Entity + Controller） | **必须** | 10 人全量审计 |
| 新增/修改 Entity 字段 | **必须** | 影响数据库结构 |
| 新增 Controller 方法 | **必须** | 影响接口安全 |
| 修改 Service 核心逻辑 | **必须** | 影响业务规则 |
| 只改 Vue 模板/样式 | 可选 | 轻量级，可跳过 |
| Bug 修复（单文件） | 可选 | 修复审计用 /debug |

### 审计执行流程

1. 读取目标代码（后端 Java + 前端 Vue3 + SQL）
2. 并行派发 10 个 AI 智能体（3 产品 + 3 研发 + 4 测试）
3. 共识判定：≥2 人检出同一问题 → P0 必改
4. 修复所有共识 P0 → 重新审计确认 → 进入 /verify

## 鹰眼团测试触发

部署完成后 AI 必须提示：
> "部署完成。需要运行鹰眼团测试吗？（gen-tests → test-api → test-e2e）"

用户确认后执行。测试失败自动进入 /debug。

## 分级测试规则（AI 自动判定）

### 轻量级（微小改动）
- 只改了 Vue 模板文字/样式、注释、`.data.ts` 列宽/标签、单个字段校验规则
→ 只跑 `/verify`，跳过回归测试

### 标准级（日常改动）
- 新增/修改 Controller 方法、Vue 组件逻辑、Service 方法、`.data.ts` Schema、≤3 个文件
→ `/verify` + `/test-api <模块>`

### 全量级（高风险改动）
- 新增/修改 Entity 字段、新增 Controller、修改 Mapper/XML/SQL、修改 pom.xml、≥5 个文件
→ `/verify` + `/test-api` + `/test-e2e` + `/test-all`

### 不变更不测试
没有 `git diff` → 全部跳过

## 推送前检查（每次 git push 前自动执行）

| 检查项 | 阻塞？ | 说明 |
|--------|:--:|------|
| boot-module 声明的模块目录存在 | 是 | 缺目录 → 编译失败 |
| system-start 有对应依赖声明 | 是 | 缺依赖 → 运行时找不到类 |
| 模块代码已 git 跟踪 | 是 | 漏提交 → CI 编译失败 |
| 铁拳团审计无共识 P0 | 是 | 共识高危必须先修 |
