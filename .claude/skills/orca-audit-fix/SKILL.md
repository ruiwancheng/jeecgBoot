# Orca 审计修复闭环 (orca-audit-fix)

## 用途

输入模块名，pi 与 Claude Code 自动交替工作：pi 修代码 → Claude 审计 → pi 修 Claude 发现的问题 → Claude 再审计，最多 3 轮，直到报告里没有 P0（共识高危）。

```
┌──────────┐    dispatch(--inject)    ┌──────────────┐
│  pi      │ ──────────────────────→ │  Claude Code │
│ (协调者)  │ ←── worker_done ────── │  (审计工人)   │
│ 开发+修复 │    (TUI 自动投递)        │  只读+写报告  │
└──────────┘                         └──────────────┘
     │                                      │
     └── 读报告提取 P0/P1 → 修代码 → 重派 ──┘
                    循环直到 0 P0 或 3 轮上限
```

## 触发条件

- 用户说"帮我审计并修复 XX 模块"、"跑审计自动修"、"/audit-fix 采购订单"
- 或 pi 完成一个开发任务后，用户想自动走审计+修复闭环

## 不适合

- 需要改数据库或外部依赖的修复（技能只改代码，部署由人工触发）
- 涉及收货/入库/财务等"改一处影响多处"的跨模块修复（仅审计，不由本技能自动修）

## 前置条件

1. Orca 应用正在运行（`orca status` 显示 ready）
2. 存在一个可用的 Claude Code 终端（`orca terminal list` 中有 title 含 "Claude" 的终端）
3. 开发环境已 commit 当前进度（便于 Claude 基于干净 diff 审计）
4. Claude Code 终端是"热终端"也可以——已验证同一终端可接受多次 dispatch

## 执行步骤

### 步骤 1：确认目标

- 用户告知模块名（如"采购订单"）和改动说明
- 如果没有明确说明，从最近一次 git diff 自动推断范围
- 确认 Claude Code 终端 handle（从 terminal list 获取，只用全柄）

### 步骤 2：第一轮审计

- 创建 task：审计目标模块的当前代码，视角 = 产品+研发+测试（单人审计模式下视角合并为一个任务）
- dispatch --inject 到 Claude Code 终端
- **等待**：TUI 自动投递 worker_done（主通道） + 每 2 分钟轮询 dispatch-show 状态（兜底）
  - ⚠️ **禁止 check --wait**（pi TUI 会自动消费编排消息，CLI 查未读必为空）
- 确定审计报告路径：`hermes/tiequan/<日期>/<模块>-audit-fix/01_审计.md`

### 步骤 3：提取并修复问题

- 读取审计报告，提取分级（P0/P1/P2/P3）
- **只修 P0、以及简单可修的 P1**（1-2 行改动、不会产生副作用的那种）
- P2/P3 及涉及多个文件的 P1 只记录、不自动修
- 每次修复后记住改动内容，用于最终汇总

### 步骤 4：复审计（如有修复）

- 如果本轮修了任何代码 → 回到步骤 2，派发第二轮审计
  任务描述："复查以下上次审计发现是否已修复，报告还剩下哪些问题。上次报告路径：`<上次路径>`"
- 如果本轮 0 修复 → 跳到步骤 5

### 步骤 5：终止判断

| 条件 | 动作 |
|------|------|
| Claude 回报 P0 = 0 | ✅ 闭环成功，输出汇总 |
| 到达第 3 轮仍有 P0 | ⚠️ 停止循环，输出汇总 + 剩余 P0 列表（人工介入） |
| Claude 终端不可用 | ❌ 降级为单体会审，老方式 |
| 单轮超时 10 分钟 | ⚠️ 标记本轮失败，继续循环 or 人工介入 |

### 步骤 6：输出汇总

格式：
```
## 审计修复闭环完成

| 轮次 | 发现 P0 | 修复 | 结果 |
|------|:--:|------|------|
| 第 1 轮 | 4 个 | 修了 2 个（补 import、加审核按钮） | 派第 2 轮 |
| 第 2 轮 | 0 个 | — | 闭环成功 ✅ |

修复内容：
1. 3 个抽屉补 import（purchase/manufacturing/sales-delivery）
2. 采购订单列表加审核按钮

遗留（人工或下个任务）：<列表>
```

## Agent 容量参考

| Agent | 最多实例 | 适用角色 |
|-------|:---:|------|
| pi | 不限 | 开发、修复、简单审计 |
| Claude Code | 1 个/工作树 | 深度审计（产品质量的分析和设计类更好） |
| Hermes | ❌ 暂不用 | 不稳定（2026-07-21 验证） |

## 等待策略（铁律）

1. **主通道**：派发后正常等待，worker_done 会自动作为编排消息出现在 pi 终端界面
2. **兜底确认**：每 2 分钟 `orca orchestration dispatch-show --task <id>` 查 `status=completed`
3. **收到回报后的下一步**：读取 payload.reportPath → 继续执行步骤 3
4. **禁止 `check --wait`**：pi 终端界面会自动投递并标记已读，CLI 再查永远为空

## 降级策略

| 情况 | 降级方式 |
|------|----------|
| Orca 不可用 | 退回到单体会审（pi 自己读代码、写报告），参考 `jeecg-tiequan-audit` 的降级模式 |
| Claude 终端不可用但 pi 实例充足 | 派给空闲 pi 终端做审计（已验证可行） |
| 无可用工人 | 完全退化为单人模式 |
| 超过 3 轮仍 0 完成 | 停止并汇总，人工接管 |

## 示例

用户说：
> "/audit-fix 销售订单明细子表"

pi 执行：
1. 确认目标 = commit da8181b 改动范围
2. 派 Claude 审计 → 4 分钟后回报 3 个 P0
3. pi 读报告 → 修 3 个 P0（补 import、税率校验、清空回填）
4. 派 Claude 复审计 → 2 分钟后回报 P0=0
5. 输出汇总：第 1 轮 3 P0 全部修复，第 2 轮 0 P0，闭环成功
