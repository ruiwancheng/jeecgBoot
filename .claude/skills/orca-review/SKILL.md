# Orca 外部评审 (orca-review)

## 用途

在 brainstorm 和 plan 阶段，pi 产出一份草案后，派给 Claude Code 做"第二意见"——从一个不同的视角审视草案，发现遗漏、矛盾或更优方案。**只评审不修改**（Claude 只写意见，pi 决定是否采纳）。

```
pi 编写草案（需求分析 or 实施计划）
     ↓
pi 派 Claude 做外部评审（dispatch --inject，附带草案全文 + 阶段视角）
     ↓
Claude 读相关代码 + 审草案 → 输出结构化评审报告
     ↓
pi 收到回报（TUI 自动投递）→ 吸收改进 → 展示最终版给用户确认
```

## 适用阶段

| 阶段 | 评审视角 | Claude 关注 |
|------|----------|------------|
| **brainstorm** | 产品/业务完整性 | 场景是否遗漏？边界是否覆盖？矛盾是否化解？数据模型是否匹配操作演示？ |
| **plan** | 架构/实施可行性 | 文件清单是否遗漏？依赖是否查证？策略是否最优？是否有高风险步骤？（对照 file-scope / code-style / 依赖查证清单） |
| **code-review 轻量** | 代码安全性 | 空值、权限、注入、并发等关键项快速扫描（非铁拳团模式，单人快速过） |

## 什么情况下用

- 用户明确说"再想想还有什么遗漏"、"帮我审查下这个方案"
- brainstorm 结论可能不完整时（复杂需求、多角色场景）
- plan 涉及高风险改动时（改多个模块、新增 Entity、修改 SQL）
- pi 自己觉得方案可能有问题时

## 什么情况下不用

- 改动很小（改一行文字 / 改个样式）—— 不需要外部评审
- 已经过了铁拳团审计的模块做小修—— 不需要再审
- verify/done 阶段—— 这些是机械检查，不是评审

## 前置条件（与 orca-audit-fix 相同）

1. Orca 应用正在运行
2. 存在可用的 Claude Code 终端（热终端复用，已验证同一终端可多次 dispatch）
3. pi 已产出待评审的草案（不要求 commit，Claude 可以读当前工作区的未提交文件）

## 执行步骤（相比铁拳团大幅简化）

### 步骤 1：准备评审输入

pi 把当前阶段的产出写成一段**结构化的评审输入**，包含：

- **背景**：什么任务、改动范围、涉及哪些模块
- **草案**：当前 brainstorm 结论 or plan 文件清单 + 策略
- **已查证项**：哪些依赖已验证通过（如菜单权限、SQL 兼容性）
- **待评审问题**：pi 自己拿不准的 2-3 个点（可选，但效果最好）

### 步骤 2：派 Claude 评审

创建 task，dispatch --inject：

```
【外部评审 - brainstorm/plan】<评审输入>
请从 <阶段视角> 审视以上草案。只读代码作为辅助，不修改任何文件。
输出格式：
## 评审结论（1 句总结）
## ✅ 思路对齐（与草案一致的部分）
## ⚠️ 遗漏或风险（草案没覆盖的、有问题的）
## 💡 优化建议（如果有更优的路径）
每个条目要具体到文件/数据/场景，禁止笼统说"缺少校验"。
报告写到 hermes/reviews/<日期>-<任务>.md。
按 preamble 回报 worker_done。
```

### 步骤 3：等待并吸收

- **等待**：TUI 自动投递 worker_done（主通道）+ 轮询 dispatch-show 兜底
- **禁止 check --wait**（与 tiequan 同样的原因：TUI 截胡消息）
- 收到回报后：读报告 → 判断哪些接受、哪些不接受 → 改进草案
- **重要**：Claude 的建议不是圣旨，pi 有权拒绝（如"该项目模式，与 XX 模块保持一致，不采纳"），拒绝的理由写进最终输出

### 步骤 4：展示给用户

把评审结果和改进后的草案一起展示：

```
Claude 外部评审结果：
  ✅ 思路对齐：XX 点已覆盖
  ⚠️ 发现 2 个遗漏：<要点>
  💡 1 个优化建议：<要点>

pi 决议：
  • 采纳：补充 XX 场景到方案中
  • 不采纳：YY 建议（与仓库模块模式一致，见 code-style.md）
  • 存疑：ZZ 建议上会讨论

改进后的最终草案如下：<完整内容>
```

## 评审报告归档

所有评审报告写入 `hermes/reviews/` 目录：

```
hermes/reviews/
├── 2026-07-22-brainstorm-销售订单物料明细.md
├── 2026-07-22-plan-编码规则绑定.md
└── ...
```

## 降级策略

| 情况 | 降级方式 |
|------|----------|
| Orca 不可用 | pi 自己扮演评审角色（从评审视角重新审视一次草案，手工写评审笔记） |
| Claude 终端不可用 | 同 Orca 不可用的降级 |
| 超时 5 分钟 | 标记本次评审失败，pi 用自己的判断提交草案给用户，标注"外部评审未完成" |

## 与铁拳团的关系

| 对比 | orca-review | 铁拳团 (jeecg-tiequan-audit) |
|------|:--:|:--:|
| 工人 | 1 人（Claude） | 10 人 |
| 时机 | 开发**前**（brainstorm/plan） | 开发**后**（审计） |
| 产出 | 评审意见 | 分级问题清单 |
| 修改权限 | pi 判定是否采纳 | 人工修复 |
| 耗时 | ~1-2 分钟 | ~5-8 分钟 |

## 评审提示词模板

### brainstorm 阶段

```
你是一位产品专家，帮我审视这个需求分析是否完整。
评审关注：
1. 正向业务流程每一步是否覆盖
2. 逆向/异常场景是否有遗漏（取消、退回、超量）
3. 上下游联动的数据口径是否一致
4. 操作演示里的动词是否都转化成数据字段
5. 是否有模糊表述需要澄清
```

### plan 阶段

```
你是一位架构评审专家，帮我审视这个实施计划。
评审关注：
1. 文件清单是否完整（对照 commit 基准检查 diff）
2. 策略判定是否正确（纯新增/覆盖标品/加字段）
3. 依赖查证是否覆盖 5 项（权限/SQL/组件/字典/父菜单）
4. 是否有更优策略（能用已有模式就不要新造轮子）
5. 是否有高风险步骤（改 Mapper SQL、改状态机）
6. 是否所有步骤都有具体文件路径和操作，而非 TODO/TBD
```
